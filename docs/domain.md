# docs/domain.md

## Purpose
このファイルには、就活管理アプリで扱う主要な概念を整理する。
実装詳細ではなく、各エンティティの役割と境界を明確にする。

---

## Core Concepts

### 1. Application
応募・選考中の企業情報を表す中核エンティティ。

主な役割:
- 企業ごとの応募状況を管理する
- Kanban 上で status を持つ
- 面談日時、メモ、求人URLなどを持つ
- 締切日や応募種別など、応募単位の情報を持つ

例:
- 企業名
- 職種名
- status
- 次回面談日時
- memo
- 求人URL
- 応募種別
- Webテスト受験状況
- 締切日

補足:
- 1つの application は 1ユーザーに属する
- application はこのアプリの中心概念である

---

### 2. Application Status
Kanban 上での進行状態を表す概念。

現在の状態:
- interest
- applied
- document_passed
- interviewing
- offer
- rejected

役割:
- ボード上の列を決める
- 進行状況を視覚化する
- DnD による更新対象になる

補足:
- 応募種別とは別概念
- Webテスト受験状況とも別概念

---

### 3. Application Type
応募の種別を表す概念。

想定:
- 夏インターン
- 本選考
- 内定者インターン

役割:
- どの種類の応募かを識別する
- status とは別軸で分類する
- フィルタや表示ラベルに使う

---

### 4. Web Test Status
Webテストの受験状況を表す概念。

想定:
- 未受験
- 受験済み
- 不要

役割:
- 受験漏れを把握する
- 未受験企業の抽出に使う

---

### 5. Deadline
応募や提出物の締切を表す概念。

役割:
- 期限管理
- 期限が近い企業の把握
- 期限切れの可視化

補足:
- 現時点では通知ではなく画面表示で扱う

---

### 6. Memo
企業ごとの自由記述メモ。

役割:
- 面談内容や補足情報を記録する
- 当面は面談テンプレの内容もここに含める

含める想定:
- 感想
- 魅力点
- 懸念点
- 次回聞きたいこと

補足:
- 今は構造化しない
- 将来的に分離が必要になれば再検討する

---

## Expanded Concepts

### 7. Reverse Offer Event
逆求人イベントでの接点情報を表す概念。
application とは別概念として扱う。

役割:
- どのイベントで話したかを記録する
- 通常応募とは異なる出会い方を管理する
- 企業との初回接点や会話メモを保持する

想定例:
- イベント名
- 開催日
- 企業名
- 担当者名
- 会話メモ
- 次アクション

補足:
- 通常応募に変わった場合、application と関連づける可能性がある
- ただし初期段階では application に無理に統合しない

---

### 8. Resume Profile
履歴書・ESで再利用するユーザー共通プロフィール。

役割:
- 定型項目の再利用
- 応募ごとの文章作成のベースになる

想定例:
- 基本情報
- 自己PR
- ガクチカ
- 研究概要
- 長期インターン経験
- ハッカソン経験

補足:
- application ごとの情報ではない
- ユーザー単位で管理する

---

### 9. Question History
企業ごとの質問履歴を表す概念。

役割:
- 聞いた質問
- まだ聞いていない質問
- 回答メモ
- 次回確認事項

補足:
- application に紐づく繰り返しデータ
- 専用エンティティとして扱うのが自然
- RLS を含めて慎重に設計する

---

### 10. Generated Email
自動生成されたメール文面を表す概念。

役割:
- 企業向けメール文面を生成する
- 生成結果を保存する
- 後で再利用・編集できるようにする

想定用途:
- 履歴書送付
- アンケート回答報告
- 期限延長お願い
- お礼メール

補足:
- 生成と保存を扱う
- 送信機能は別概念として分けて考える余地がある

---

## Boundary Rules
- status は進行状態のみを表す
- application_type は応募の種類を表す
- web_test_status は Webテスト受験状況を表す
- reverse_offer_event は application と別概念
- resume_profile はユーザー共通情報であり application と別概念
- question_history は application に紐づく子エンティティ
- generated_email は応募支援の生成物として扱う

---

## Modeling Principles
- 1つのフィールドに複数の責務を混ぜない
- 既存 application に無理に新概念を押し込まない
- 一覧・抽出したくなるものは、早めに独立概念として検討する
- ただし初期段階では実装コストとのバランスを優先する
