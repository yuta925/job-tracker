# docs/roadmap.md

## Purpose
このファイルには実装の順番を記録する。
仕様の詳細ではなく、どの順で進めるかを整理する。

---

## Overall Strategy
方針は以下の順とする。

1. 既存不具合を先に直す
2. 既存 application に自然に乗る軽量機能を追加する
3. 別概念や新規エンティティが必要な機能に進む
4. 外部連携や生成系を最後に扱う

---

## Phase 0: Decision Alignment
実装前に以下の前提を採用済み。

- 日時はローカル時刻として扱う
- 締切日の自動リマインドは画面表示のみ
- 面談メモテンプレは memo に含める
- 逆求人イベント記録は別概念として扱う
- メール文面生成は保存もする

---

## Phase 1: Bug Fixes
### 1. 面談日時ズレ修正
目的:
- 入力した日時が、保存後も同じ日時で表示されるようにする

対象:
- フォーム入力
- 保存処理
- 表示処理
- タイムゾーンのラウンドトリップ確認

### 2. 求人URL導線追加
目的:
- 登録済みの求人URLへカードから遷移できるようにする

対象:
- カード表示
- URL がある場合のみ表示
- 外部リンクとして安全に開く

---

## Phase 2: Lightweight Application Extensions
### 3. 応募種別分類
- 夏インターン / 本選考 / 内定者インターンの分類を追加する
- status とは別軸で管理する
- 入力・表示・フィルタまでを対象にする

### 4. Webテスト受験状況の管理
- Webテスト受験状況を入力できるようにする
- 未受験企業を抽出できるようにする

### 5. 締切日の追加
- 締切日を保存できるようにする
- 期限が近い / 期限切れの視覚表示を行う
- 通知処理は含めない

### 6. 面談メモテンプレ
- 面談後の記録をテンプレ入力しやすくする
- 保存先は memo とする

---

## Phase 3: Domain Expansion
### 7. 逆求人イベント記録
- 通常応募とは別の概念として管理する
- まずは最小単位の情報整理から着手する
- application とどう接続するかを明確にして進める

### 8. 履歴書の定型項目の再利用
- ユーザー共通プロフィールとして管理する
- 自己PR、ガクチカ、研究概要などの再利用を目的とする

---

## Phase 4: New Entity / Higher Complexity
### 9. 企業ごとの質問履歴管理
- 企業ごとの質問・回答・次回聞きたいことを管理する
- application 配下の繰り返しデータとして扱う
- 新規テーブル、アクセス層、UI、RLS を含む

---

## Phase 5: Generation Features
### 10. メール文面の自動生成
- メール文面を生成する
- 生成結果を保存する
- まずは送付・お礼・期限延長などの定型用途を優先する

---

## Execution Rules
- 各 Phase は小タスクに分けて進める
- 各タスクで lint / typecheck / test の通過を確認する
- バグ修正と機能追加を同時に大きく混ぜない
- 変更対象が増えたら、先に影響範囲を確認してから進める
